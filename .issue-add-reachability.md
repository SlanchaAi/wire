## Summary

`wire add <nick>@<domain>` reports `status: drop_sent` and pins the peer at `PENDING_ACK` even when the local agent is **not federation-reachable** — i.e. it has no claimed handle on that relay and its primary `self` slot is an unreachable loopback (`http://127.0.0.1:…`). The pair-intro delivered to the peer carries a `self` endpoint pointing at the local agent's **loopback** address, so the peer can never deliver `pair_drop_ack` back. The pairing then sits at `PENDING_ACK` forever with no error — it looks initiated but can never complete "via relay." Reported as "you are pairing locally with them, not via relay."

## Environment

- `slancha-wire` 0.16.0, macOS
- Local agent `saffron-burrow` whose `self` slot was `http://127.0.0.1:18791` (a local relay, also down), scope recorded as `federation` (see Related #2 — wrong scope for a loopback URL).
- Peer `raven-kettle@wireup.net`, correctly claimed + resolvable on the federation relay.

## Repro

1. Have a local agent whose only `self` endpoint is a loopback slot (e.g. fresh `wire init` / local-first setup) and who is **not** claimed on `wireup.net`.
2. `wire add raven-kettle@wireup.net --json`

## Actual

```json
{"status":"drop_sent","peer_handle":"raven-kettle","paired_with":"did:wire:raven-kettle-465c3352"}
```

- Peer is pinned `PENDING_ACK`.
- The intro the peer receives advertises the local agent's **loopback** `self` endpoint — unreachable from another host.
- The local agent is **not** in the relay's handle directory:
  `curl https://wireup.net/.well-known/wire/agent?handle=saffron-burrow` → `HTTP 404`.
- `pair_drop_ack` can never arrive → permanent silent `PENDING_ACK`.

## Expected

`wire add <nick>@<domain>` (federation) should ensure the local agent is actually reachable on that relay before (or while) delivering an intro, OR refuse/warn loudly. Concretely, at least one of:

- **Pre-flight check:** before delivering a federation intro, verify the local agent has a non-loopback `self` endpoint on a federation relay AND is claimed there; if not, error with a remediation hint (`run 'wire up --relay <domain>'` / `wire bind-relay <domain>` + `wire claim`).
- **Don't advertise a loopback `self` endpoint** in a *federation* pair-intro — select a federation-scoped self endpoint; if none exists, that's the error above.
- At minimum, **WARN** in the `wire add` output + JSON (`reachable_self: false`) so the operator knows the peer cannot ack them.

## Workaround (what actually completed registration)

```bash
wire bind-relay https://wireup.net --replace --migrate-pinned   # make wireup.net the PRIMARY self slot
wire claim saffron-burrow --public-url https://wireup.net        # now lands on wireup.net (see Related #2)
curl -s "https://wireup.net/.well-known/wire/agent?handle=saffron-burrow"   # HTTP 200 — now registered
wire add raven-kettle@wireup.net                                 # intro now advertises the reachable slot
```

## Related

1. `wire claim --relay <URL>` is ignored when the relay isn't already the primary self slot (separate issue) — it routes to the primary slot's relay. That's why the workaround needs `bind-relay --replace` *before* `claim`.
2. `bind-relay` recorded scope `federation` for a `http://127.0.0.1:…` (loopback) self endpoint; loopback should infer scope `local`. A correct scope would let `wire add` distinguish "I have a federation self slot" from "I only have a local slot."

## Severity

High (UX/correctness) — federation pairing silently never completes for any local-first / un-claimed agent, with a success-looking `drop_sent` + `PENDING_ACK`. The operator gets no signal that they aren't reachable.
