# GB10 DGX Spark — wire 0.14.1 → 0.16.0 upgrade notes (2026-06-14)

Operator field notes from upgrading `wire` (slancha-wire) on two GB10 DGX Sparks
(`promaxgb10-d325` + `spark-472e`), `linux-systemd-user`, aarch64. Three reproducible
bugs were found and filed upstream; the systemd-side workarounds applied on these
boxes are recorded here for posterity.

## Filed issues

- **#274** — `wire upgrade` writes a literal ` (deleted)` token into the `ExecStart=`
  of every refreshed systemd unit. Root cause: the unit refresh derives the binary
  path from `/proc/self/exe` *after* `cargo install` replaces the running binary
  in place, and the kernel reports the unlinked exe as `…/wire (deleted)`. Result:
  `error: unrecognized subcommand '(deleted)'`, unit flaps forever. **High severity** —
  every cargo-path `wire upgrade` leaves the daemon/supervisor dead.
- **#275** — `wire upgrade --refresh-stale-children` kills every `wire daemon`, but
  the `--all-sessions` supervisor only respawns sessions in the registry
  (`registry.json` by_cwd). Free-floating per-session daemons started with an explicit
  `WIRE_HOME=…/by-key/<hash>` are killed and never respawned → that identity silently
  stops syncing its relay slot, despite the upgrade output claiming it will respawn.
- **#276** — `wire upgrade` PATH-shadow check false-positives when the active PATH
  entry is a *symlink* to the upgraded binary (compares raw paths, not resolved
  targets / sha256). Misleads right after the user applies the tool's own recommended
  symlink fix.

## Workarounds applied on these boxes

1. **`(deleted)` ExecStart (#274):**
   ```sh
   sed -i 's# (deleted) # #g' ~/.config/systemd/user/wire-daemon.service \
                                ~/.config/systemd/user/wire-local-relay.service
   systemctl --user daemon-reload && systemctl --user restart wire-daemon
   ```
2. **PATH shadow (#276):** `~/.local/bin/wire` (install.sh target, PATH-first) shadowed
   `~/.cargo/bin/wire` (0.15/0.16). Fixed once with a symlink so both PATH entries and
   every unit pinning the absolute `~/.local/bin/wire` path follow future cargo upgrades:
   ```sh
   ln -sf ~/.cargo/bin/wire ~/.local/bin/wire
   ```
3. **Free-floating session daemon (#275):** relaunch the killed per-session daemon
   directly on the new binary:
   ```sh
   WIRE_HOME=~/.local/state/wire/sessions/by-key/<hash> \
     setsid nohup ~/.local/bin/wire daemon --interval 5 >/tmp/x.log 2>&1 </dev/null &
   ```
4. **`wire-public-relay.service`** was pinned to a stale `~/Source/wire/target/release/wire`
   source build (0.14.1) → repointed `ExecStart` to `~/.local/bin/wire` and restarted
   (localhost:8770 behind a cloudflared tunnel; verified local `/healthz` + the tunnel
   both return 200 after).
5. **`wire-local-relay.service`** (Rust, `--local-only --bind 127.0.0.1:8771`) perma-collides
   with a `python3 -m http.server 8771` landing page that owns :8771 → "Address already
   in use", infinite flap. Disabled it (`systemctl --user disable --now wire-local-relay`);
   sessions here pair via the wireup.net relay, so the local :8771 relay was unused. Real
   fix if it's ever wanted: move one service to a distinct port (:8771 is wire's hardcoded
   local-relay default that `wire session new` probes).

## Verification after upgrade

- `wire upgrade --check` → `cli/latest/pidfile` all 0.16.0, no stale, no PATH warn.
- `which -a wire` → both entries resolve to the same 0.16.0 binary (identical sha256).
- All session daemons live on 0.16.0; `wire-daemon` + `wire-public-relay` units active;
  relay `/healthz` + tunnel both 200.

## Follow-up idea (from triage with peer `lavender-shore`)

A `wire status --wait-daemon-running --timeout <secs>` built-in would let bringup scripts
drop hand-rolled spin-loops (which are easy to get wrong — e.g. a `pgrep -f` wait that
self-matches its own shell's command line and kills it). One well-behaved primitive beats
every host reinventing the wait. Cousin of #275.
