# cli.rs Module Split Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Split the 15,667-line `src/cli.rs` monolith into a `src/cli/` module tree grouped by command family, with zero behavior change.

**Architecture:** `src/cli/mod.rs` keeps the clap surface (`Cli`, `Command`, all subcommand enums) plus `run()` dispatch and shared helpers; each command family's `cmd_*` handlers move to a sibling module. All moves are pure mechanical relocation — same fn bodies, same signatures, visibility widened to `pub(crate)` only where cross-module calls require it, `pub use` re-exports in `mod.rs` preserve every existing `crate::cli::X` path so no caller outside `src/cli/` changes.

**Tech Stack:** Rust 2024 / cargo. Gate per task: `cargo fmt --all -- --check && cargo clippy --all-targets -- -D warnings && cargo test --all-targets -- --test-threads=1` (run each bare, check each exit code; final full gate in `test-env/run.sh`).

---

## Ground rules (apply to every task)

- **Pure relocation.** Copy fn/struct/test-mod text verbatim. No logic edits, no renames, no signature changes, no "while I'm here" cleanups.
- **Visibility:** a fn that stays called only within its new module keeps `fn`; a fn called from another `cli/` module or elsewhere in the crate becomes `pub(crate) fn`. Already-`pub` items stay `pub` and get re-exported from `mod.rs` (`pub use`).
- **External callers must not change.** `src/lib.rs`, `src/mcp.rs`, `src/send.rs`, `src/bin/wire.rs`, tests/ refer to `crate::cli::{run, Cli, run_sync_push, run_sync_pull, maybe_auto_init_cwd_session, closest_candidates, error_smells_like_slot_4xx, ...}` — `mod.rs` re-exports keep all of these resolving. Verify with `cargo build` before running the full gate.
- **Embedded `#[cfg(test)]` modules move with the code they test** (see manifest below for which test block belongs to which family).
- **Commit after each green gate.** One module per commit: `refactor(cli): extract <module> (mechanical move, no behavior change)`.
- If a move turns out to need a logic change to compile (it shouldn't), STOP — that's a signal the grouping is wrong; re-group instead of editing logic.

### Task 0: Scaffold `src/cli/` (git mv preserves history)

- [ ] **Step 1:** `git switch -c refactor/cli-split` (branch from main AFTER workstream-A fix branches merge, or rebase later)
- [ ] **Step 2:** `git mv src/cli.rs src/cli/mod.rs`
- [ ] **Step 3:** `cargo build` — expect PASS (path change only; `mod cli;` in lib.rs resolves `cli/mod.rs`)
- [ ] **Step 4:** Full gate (3 bare commands, check each `$?`) — expect PASS
- [ ] **Step 5:** `git commit -m "refactor(cli): move cli.rs to cli/mod.rs (scaffold, no code change)"`

### Tasks 1–11: Extract one family per task, in this order

Order picked to start small (lifecycle ~260 lines) and end with the big, entangled ones (session, group):

1. `lifecycle.rs` 2. `upgrade.rs` 3. `setup.rs` 4. `status.rs` 5. `comms.rs` 6. `relay.rs` 7. `pairing.rs` 8. `group.rs` 9. `mesh.rs` 10. `session.rs` 11. `identity.rs`

Per-task steps (identical recipe, swap the module name):

- [ ] **Step 1:** Create `src/cli/<family>.rs` with the family's fns/structs/test-mods cut verbatim from `mod.rs` (use the manifest). Add `use` lines the new file needs (copy from mod.rs head; prune unused at the end of the task, NOT mid-move).
- [ ] **Step 2:** In `mod.rs`: add `mod <family>;`, plus `pub use <family>::{...}` for items external code references; change dispatch arms to `<family>::cmd_*(...)` (or rely on re-exports — prefer explicit module paths in the dispatch for readability).
- [ ] **Step 3:** `cargo build` — fix ONLY visibility (`pub(crate)`) and import errors. Expect PASS with no logic edits.
- [ ] **Step 4:** Full gate, 3 bare commands, each exit checked — expect PASS.
- [ ] **Step 5:** `git add src/cli && git commit -m "refactor(cli): extract <family>.rs (mechanical move, no behavior change)"`

### Task 12: Final sweep

## Explicitly out of scope

- Any DRY/dedup pass (repeated home-resolution, relay-client construction) — that's a FOLLOW-UP after the mechanical split lands, never mixed into a move commit.
- RFC-006 anything.
- Touching `src/mcp.rs`, `src/send.rs`, or any caller beyond what re-exports require (target: zero caller edits).
