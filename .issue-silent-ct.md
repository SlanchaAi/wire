## Summary

When `wire tail` receives an encrypted DM (`kind=1001`, `enc=wire-x25519.v1`) that the running binary **cannot decrypt** — e.g. a stale binary built before DM encryption landed — it prints the raw ciphertext blob as the body, with a green `sig ✓`, and **no warning**. The operator sees a verified-signed message whose body is gibberish JSON (`{"ct":"AkSb…"}`) and gets zero signal that there is readable plaintext they're missing. Verified signature + unreadable body + no warning = silently flying blind on encrypted DMs.

## Environment

- Receiver: macOS, **two binaries on one host** — installed `wire 0.14.1` (predates D1 DM encryption) and a freshly-built `wire 0.16.0`.
- Sender: `raven-kettle` on `wire 0.16.0` (Windows MSVC release), sending `enc=wire-x25519.v1` DMs.

## Repro

1. Receive an `enc=wire-x25519.v1` DM from a peer (any peer on a build with D1 DM encryption).
2. Tail the thread with a binary that lacks DM decryption (here, the installed `0.14.1`):

```
$ wire --version
wire 0.14.1
$ wire tail raven-kettle
[2026-06-14T17:28:31Z did:wire:raven-kettle-465c3352 kind=1001 claim] {"ct":"AkSbz2hEQ8Fi…=="} | sig ✓
```

3. Tail the same thread, same `$WIRE_HOME`, with `0.16.0`:

```
$ ./wire --version
wire 0.16.0
$ ./wire tail raven-kettle
[2026-06-14T17:28:31Z did:wire:raven-kettle-465c3352 kind=1001 claim] Got it, verified end-to-end. … | sig ✓
```

Same event, same key material. `0.16.0` decrypts; `0.14.1` shows the ciphertext as the body with no indication it's encrypted-and-unreadable.

## Expected

A binary that cannot decrypt an `enc=<discriminator>` body it doesn't support (unknown/unsupported `enc`, or a decrypt failure) should render a clear placeholder + warning instead of the raw ciphertext, e.g.:

```
[… kind=1001 claim] <encrypted DM: enc=wire-x25519.v1 — this wire build cannot decrypt it; run `wire upgrade`> | sig ✓
```

and surface it in `--json` (e.g. `decryptable: false`, `enc: "wire-x25519.v1"`) so tooling can flag it. The signature is genuinely valid, so `sig ✓` is correct — but the body must not be presented as if it were the plaintext.

## Notes

- The trigger here is a **stale binary** (a real situation — `wire upgrade` swaps the daemon but not always the binary the operator's shell resolves; see the broader stale-binary class). But the same gap applies to any future `enc` discriminator a given build doesn't understand — forward-compat for the encryption envelope.
- Distinct from a decrypt *error* on a supported build (wrong key / corrupt ct), which should also warn rather than dump ciphertext.

## Severity

Medium — silent confidentiality/UX failure. An operator on a stale binary believes they've read a peer's verified message when they've actually read gibberish; no prompt to upgrade.
