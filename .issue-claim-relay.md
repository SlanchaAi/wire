## Summary

`wire claim <nick> --relay <URL>` ignores `--relay` and POSTs the claim to the relay of the agent's **primary `self` slot**, not the `--relay` target. If the primary self slot is a different (or unreachable) relay, the claim fails or lands on the wrong relay — you cannot claim your handle on a federation relay you aren't already primarily bound to.

`wire claim --help` documents `--relay` as "Relay to claim the nick on" — so the flag is silently non-functional.

## Environment

- `slancha-wire` 0.16.0, macOS
- Agent `saffron-burrow` with primary `self` slot on `http://127.0.0.1:18791` (down), plus an additive federation slot on `https://wireup.net`.

## Repro

```bash
# self primary slot = http://127.0.0.1:18791 ; additive slot exists on https://wireup.net
wire claim saffron-burrow --relay https://wireup.net --public-url https://wireup.net --json
```

## Actual

```
error: POST http://127.0.0.1:18791/v1/handle/claim: error sending request ... tcp connect error: Connection refused
```

It POSTed to `127.0.0.1:18791` (the primary self slot's relay) despite `--relay https://wireup.net`.

Only after making wireup.net the primary slot (`wire bind-relay https://wireup.net --replace`) did a plain `wire claim saffron-burrow` (no `--relay`) correctly hit wireup.net and return `status: claimed`.

## Expected

`--relay <URL>` selects the relay the claim is POSTed to (allocating/using the self slot on that relay), independent of which slot is primary. Or, if claiming requires a slot on the target relay, claim should use the existing `self.endpoints[]` entry for `<URL>` rather than the primary slot.

## Severity

Medium — blocks claiming on a federation relay for any agent whose primary slot is elsewhere; forces a destructive `bind-relay --replace` as a prerequisite. Contributes to the federation-pairing-reachability gap (separate issue).
