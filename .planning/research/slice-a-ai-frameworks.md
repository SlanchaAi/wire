# Slice A — Wire positioning vs AI agent frameworks, A2A protocols, memory protocols

**Date:** 2026-05-21
**Wire baseline:** v0.6.10 (May 2026). Ed25519-signed federated message bus, DIDs (`did:wire:<handle>-<hex>`), HTTP+mailbox relay (`wireup.net`) + local-relay loopback transport, bilateral pair gate (receiver consent required), v0.6 mesh primitives (`pair-all-local`, `mesh status/broadcast/role/route`), MCP server exposing `wire_send/wire_tail/wire_peers/wire_mesh_*`. LOCKED v0.7+: identity-first lifecycle (anon→local→federation), two bounded transports (UDS + HTTP+relay), harness-agnostic via `WIRE_AS` env + `wire launch --as <tag>`. Trust prior format: `[tier, source, age]`, P=primary, S=secondary, T=tertiary.

**Reading guide.** Three categories below differ in their relationship to wire. Category 1 (orchestration frameworks) is mostly *complementary* — they orchestrate *one process's* multi-role decomposition; wire is the inter-process / inter-machine layer underneath. Category 2 (A2A protocols) is the *direct comparable* and the most important — Google A2A in particular is the convergent competitor. Category 7 (memory) is *orthogonal* — different problem, plausible integration target.

---

## Assumptions surfaced

- **A1 — Wire's "bilateral consent" is differentiated.** Google A2A and IBM ACP both delegate auth/consent to implementers; neither bakes in a default "receiver must accept before write-access" gate. I treat that as a wire-unique posture, but it depends on the reading that no production A2A deployment yet ships an opinionated consent default. If a vendor-shipped A2A reference implementation already does this by default, the differentiation softens.
- **A2 — "Federated" means different things.** Wire is mailbox-federated (Mastodon-shaped: `nick@domain`, WebFinger-style resolution, per-relay sovereignty). A2A protocol is *interop-federated* (standard wire format across vendors, but no shared identity / no mailbox / no built-in directory). I read these as orthogonal axes, not directly competing definitions.
- **A3 — MCP is not an A2A protocol.** Despite its name and the Anthropic donation to the Linux Foundation Agentic AI Foundation (Dec 2025), MCP is host↔server (one-direction tool invocation), not agent↔agent peer-to-peer. Some research surfaces (`mcp_agent_mail`, CA-MCP papers) wedge agent-to-agent semantics on top of MCP, but the base protocol is asymmetric. If MCP gains a symmetric peer transport in 2026 spec revisions, this assumption needs re-checking.
- **A4 — Sister-agent same-machine coordination is a category wire is shaped for.** None of the surveyed orchestration frameworks treat "two separate Claude Code processes in two cwds" as a first-class case — they treat one process with N internal agents. Wire's `--local-only` + `pair-all-local` recipe targets a use case the orchestration frameworks don't even articulate.

## Category 1 — Multi-agent AI orchestration frameworks

These are *intra-process* frameworks: one program, an LLM, and a library coordinating multiple roles, tools, handoffs, and subgraphs. They are not transports. The unit of execution is "agent" but the unit of deployment is "process."

Risk: Pydantic AI and a few others have already integrated A2A — if A2A becomes the lingua franca, wire competes on the transport layer, not the orchestration layer.

The "wire MCP server presents wire peers as A2A agents to non-wire clients" play is plausible and high-leverage; doc it as a v0.7+ direction.

## Category 7 — AI-to-AI memory + handoff protocols

Memory is a different problem from messaging. These projects store an agent's *state across sessions*; wire moves *messages between agents*. They are orthogonal at design level but compete for the same mental slot in operators' heads ("how do I make my agents persistent?").

**Reading.** Memory is not wire's problem. But the mental model "agents that persist" overlaps with the mental model "agents that coordinate," and operators conflate the two. Wire should *not* try to solve memory — it should plug into Letta / mem0 / Zep as the messaging substrate beneath them. The mem0 + Claude Code MCP integration is a worked example of how this composes: an operator running wire + mem0 inside Claude Code gets messaging + memory without wire ever having to ship a memory layer.

## Cross-category patterns

**1. The category is converging on three layers.** Tool-integration (MCP), agent-to-agent transport (A2A + competitors), and orchestration (LangGraph / CrewAI / AutoGen / ADK). Memory (Letta / mem0 / Zep) is a side-quest most agents acquire as a fourth layer. Wire sits cleanly in the A2A transport layer — the one with the least settled standard and the most fragmented vendor matrix.

**2. Bilateral consent is wire's most under-marketed wedge.** Every other A2A protocol punts auth/consent to implementers ("A2A leaves authentication to implementers and relies on agent opacity for IP protection" — Semgrep, 6mo). Wire's `pair-accept` gate is a *default opinionated* posture — strangers cannot write to your inbox without your acknowledgement. In a world about to be flooded with spam-grade auto-agents, that default is a moat. Worth promoting from implementation detail to top-line positioning.

**3. Mailbox-async is wire's second under-marketed wedge.** A2A is request/response — both agents must be live. Wire is mailbox — sender ships event, recipient picks up when next online. That property is the difference between "two agents on always-on servers" and "one Claude Code on my laptop that I close every night." If wire's ICP is *operator-owned agents*, not *cloud-hosted always-on agents*, mailbox-async is the structural answer.

**4. The same-machine sister-agent UX is a category nobody else articulates.** None of the orchestration frameworks, A2A protocols, or memory layers treats "two Claude Code processes in two cwds on one laptop" as a first-class scenario. Wire's `--local-only` + `pair-all-local` recipe (v0.6.6+) is the only design I found that answers that case end-to-end. This is wire's most-deep moat for individual-operator ICP — but it's also why wire isn't yet on enterprise procurement radars (those buyers want the cross-cloud A2A story).

**5. The realistic 12-month threat is Google A2A becoming load-bearing.** ACP folded in. Salesforce uses it. Pydantic AI ships an A2A client. ADK ships an A2A endpoint. If every meaningful agent already has an A2A endpoint, wire's "you need a transport between agents" pitch dilutes. The right defensive move (and the right offensive move) is the same: **wire speaks A2A on the outside, while keeping its own identity / consent / mailbox semantics on the inside.** That makes wire an A2A node with stronger defaults — not a competitor to A2A.

**6. Memory is best left to memory.** Wire should resist building memory features. Letta / mem0 / Zep are well-funded and well-shaped for that problem. Wire's job is to be the bytes between agents, not the bytes inside one. The integration story (`wire send` → `mem0.add()` → searchable by recipient agent) is the right level of involvement.

**7. The identity-first v0.7 direction lines up with the field.** NANDA's DID-based discovery + verifiable credentials + reputation arc is the same direction wire's LOCKED v0.7+ vision points. Wire is much further along on shipping; NANDA is further along on the long-horizon trust-fabric design. Worth tracking NANDA's published artifacts as a "what wire might look like in 2028" signal, without copying the scope.

**8. Honest weakness.** Wire is solo-maintained at v0.6.10 with a public-good `wireup.net` relay at $0/mo of Fly.io spend. A2A has 150 orgs, MCP has the Linux Foundation. The standards-tier competitors have *governance capital* wire does not. The mitigation isn't more code — it's adopting A2A's wire format for cross-vendor interop so wire interoperates cheaply rather than competes expensively.
